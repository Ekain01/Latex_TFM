\section{Introducción al entorno de SIMATIC WinCC V8.1}
SIMATIC WinCC V8.1 es un sistema \ac{SCADA}, basado en los sistemas operativos Microsoft Windows, desarrollado por Siemens AG. Está diseñado para la visualización, control y operación de procesos, flujos de producción, máquinas y plantas críticas en diversos sectores industriales.

\par \indent WinCC actúa como una \ac{HMI} de alto nivel. A diferencia de los paneles de operador básicos ubicados a pie de máquina, un \ac{HMI} de alto nivel como WinCC proporciona un entorno gráfico avanzado que permite a los operadores del centro de control manejar, monitorizar y entender de forma intuitiva los procesos dinámicos que se ejecutan en la planta sin tener que estar en la ubicación de la máquina. La comunicación entre el entorno de supervisión WinCC y la maquinaria física se efectúa de manera bidireccional a través de sistemas de automatización, empleando protocolos de comunicación industrial.(\ref{fig:wincc_interface})

\par \indent Por otro lado, WinCC es plataforma altamente programable. Integra un motor de ejecución que permite la creación de lógicas complejas y automatización de tareas de supervisión utilizando interfaces de aplicación  como \ac{ANSI-C} y \ac{VBS}. Por otro lado, WinCC V8.1 utiliza de forma nativa el motor de bases de datos Microsoft SQL Server. Este gestor de bases de datos opera en segundo plano y se encarga del archivado histórico continuo de las variables del proceso y del registro de eventos y fallos.

\par \indent Desde el punto de vista de la arquitectura de software, el entorno de WinCC se divide en dos componentes fundamentales y complementarios: el sistema de configuración o \ac{CS} y el entorno de ejecución o \ac{RT}.

Estos subsistemas se encargan de diferentes tareas dentro del sistema SCADA, como la creación de variables, la configuración de comunicaciones, la gestión de alarmas, la creación de informes, entre otras funciones. \subsection{Software de ejecución (RT) WinCC Runtime}
El software de ejecución WinCC Runtime (o \ac{RT}) es el motor  motor encargado de administrar todos los subservicios de WinCC. Es el encargado de ejecutar la adquisición de datosde los PLCs en tiempo real, actualizar dinámicamente la interfaz gráfica, procesarla lógica de los scripts, gestionar el control de acceso de los usuarios y archivar de manera segura los eventos y datos históricos en las bases de datos SQL.\par
\section{Arquitecturas de software de supervisión en SIMATIC WinCC V8.1}
La arquitectura en SIMATIC WinCC representa la repartición de los componentes \ac{CS} y \ac{RT} de software sobre los recursos computacionales. Dependiendo de la topología de la red, el número de estaciones de operación y los requisitos de redundancia, se pueden implementar diferentes arquitecturas de software. Estas arquitecturas pueden ser monopuesto o multipuesto.\par
\subsection{Sistemas Monopuesto (\textit{Single User Systems})}
En la arquitectura de sistema monopuesto un único equipo físico o estación se encarga de realizar las tareas relacionadas al software de configuración y ejecución. Es decir, el equipo se encarga de realizar por un lado las labores de ingeniería (Diseño de interfaces gráficas, gestion de variables etc.) y por el otro realiza las comunicaciones con la parte operativa (mediante los protocolos de comunicación como Profinet, ModBus u \ac{OPC UA} etc.),  ejecuta la lógica de \textit{scripts}, almacena los históricos y alarmas en la base de datos SQL a la vez que muestra la interfaz gráfica al operario en los monitores. \par
Siemens permite dos arquitecturas para este tipo de sistemas: La primera, una estación única que integra configuración y \ac{RT} en una única estación (Figura \ref{fig:monopuesto_simple}), y  la segunda, una variante donde un PC configurado con redundancia ejecuta el software \ac{RT} por si la estación principal falla. (Figura \ref{fig:monopuesto_redundante}).

\subsection{Sistemas Multipuesto (\textit{Multy User Systems})}
Un sistema multipuesto es una arquitectura distribuida diseñada para separar las tareas de procesamiento de datos de las tareas de visualización. A diferencia de un sistema monopuesto (donde una única estación de operación realiza todas las tareas), en un sistema multipuesto la carga de trabajo se divide entre uno o varios Servidores centrales (configuradas como \textit{Multi-User Project}) y múltiples estaciones Cliente (configuradas como \textit{Client Project}). Este enfoque permite una variedad grande de arquitecturas y configuraciones de proyecto, adaptándose a las necesidades específicas de cada instalación, siendo el foco de implementación final. (Figura \ref{fig:multipuesto_distribuido}) \par 

\section{Virtualización mediante \ac{SIVaaS}}
Las arquitecturas de sistemas WinCC  pueden ser implementadas de forma física o virtualizada.
La virtualización es un método por el que los recursos físicos de un equipo se pueden dividir en diversos entornos virtuales. Para ello se utiliza un hipervisor que se encarga de ejecutar múltiples \ac{VM} aisladas, compartiendo recursos como \ac{CPU}, memoria y almacenamiento. Este aislamiento evita los conflictos debidos a las dependencias de software y proporciona la capacidad de iniciar y detener las máquinas virtuales de forma independiente. Solo la máquina virtual en cuestión se ve afectada por un fallo (o caída del sistema). Existen dos tipos de hipervisores: Alojados y Metal-Desnudos.
\subsection{Hypervisor alojado  (\textit{Hosted Hypervisor})}
Este tipo se instala como una aplicación de software más dentro de un sistema operativo convencional que ya está funcionando en el equipo físico (conocido como \ac{OS} Anfitrión o Host \ac{OS}).\par

Este tipo de hipervisor genera una dependencia del \ac{OS} base ya que el hipervisor debe pedir constantemente permiso para acceder a los recursos del sistema lo cual genera un cuello de botella disminuyendo el rendimiento. \par

Por otro lado, esta dependencia del \ac{OS} base es una vulnerabilidad en sí misma para la seguridad y ciberseguridad del sistema, ya que un malfuncionamiento o un virus podrían contagiarse a cada una de las instancias de maquinas virtuales que corren encima.

\subsection{Hypervisor metal desnudo (\textit{Bare-Metal Hypervisor})}

Se le denomina \textit{Bare-Metal} (metal desnudo) porque se instala y ejecuta directamente sobre el hardware físico del servidor. No hay ningún sistema operativo tradicional (como Windows Server o Linux) actuando como intermediario. El hipervisor en sí mismo es un sistema operativo altamente especializado y minimalista diseñado con un único propósito: gestionar y asignar recursos físicos a las máquinas virtuales. Al no tener que atravesar ningún sistema operativo anfitrión, las maquinas virtuales tienen acceso más directo a los recursos del sistema reduciendo la latencia y sobrecarga. \par

Al contrario que un hipervisor alojado, es seguro ante vulnerabilidades, debido a que que tiene una superficie de ataque reducida. Además, las máquinas virtuales están aisladas por lo que si una maquina virtual se ve comprometida por un malware el resto de maquinas permanecerán seguras.\par

\subsection{Virtualización SIMATIC como Servicio (\ac{SIVaaS})}
Siemens ofrece una solución de virtualización completa para sistemas de control basada en hipervisores bare-metal llamada Virtualización SIMATIC como Servicio (\ac{SIVaaS}). \ac{SIVaaS} es una solución de virtualización completa para sistemas de control basados en PC. El hardware, el software, la configuración y los servicios de ciclo de vida están totalmente integrados, abarcando desde la configuración del host y el hipervisor hasta las máquinas virtuales, los sistemas operativos y la instalación de software llave en mano. \par

La Sala de Control Central \ac{MCR} requiere de un entorno de hardware y computación de alta disponibilidad capaz de asegurar la ejecución ininterrumpida del SCADA.\par 

Durante este proyecto se abordará únicamente la solución basada en una arquitectura monopuesto. Esta arquitectura puede ser posteriormente convertida a multipuesto virtualizada o física sin necesidad de agregar nuevos recursos computacionales, simplemente redistribuyendo las tareas de configuración y ejecución entre los diferentes hosts y máquinas virtuales. \par

Como se describe en el anexo \ref{cap:Pruebas}, la \ac{MCR} dispone de routers que asignan direcciones \ac{IP} públicas y privadas al \ac{SCADA}, facilitando tanto la comunicación con los sistemas de campo como la transferencia de datos a plataformas de análisis en la nube. Asimismo, cada subestación de transformación e interconexión cuenta con un router que proporciona direcciones \ac{IP} privadas y switches a nivel de enlace hacia los aerogeneradores. De tal forma que los aerogeneradores que cuelguen de una misma subestación comparten la misma dirección \ac{IP}.\par
\section{\ac{IM} de la solución SCADA}
\label{cap:modelo_informacion}
El modelo de información de un sistema es la representación estructurada de los datos y la información que maneja ese sistema. Con ello se define la forma en que se organizan, almacenan y acceden a los datoa dentro del sistema. El entorno de SIMATIC WinCC V8.1 permite la creación de variables que se utilizan para representar los datos del proceso, las alarmas, los eventos y otros aspectos operativos del sistema SCADA. Estas variables pueden ser internas, externas o de estructura, dependiendo de su función y origen dentro del sistema. \par

 Las variables internas son aquellas que se crean y gestionan dentro del entorno de WinCC, únicamente consumen memoria \ac{RAM} y no estan limitadas por las licencias de Siemens. Mientras que las variables externas son aquellas que se adquieren a través de un canal de comunicaciones con los dispositivos físicos o sistemas externos. Estas variables externas consumen licencias de Siemens por lo que es conveniente limitar su uso en base a la licencia disponible. Las variables de estructura, por otro lado, son variables orientadas a objetos las cuales sirven para estructurar la información. Este tipo de variables son muy útiles para organizar y replicar la información de forma lógica y coherente. Además se pueden configurar como externas o internas segun las necesidades especificas.\par
\subsection{Variables internas}
El entorno de SIMATIC WinCC V8.1 ofrece una lista de variables internas predeterminadas que se utilizan para diversas funciones dentro del sistema SCADA. Estas variables internas son fundamentales para la operación del sistema, ya que proporcionan información crítica sobre el estado del sistema, el usuario actual, la máquina local, el idioma activo, la hora local, entre otros aspectos operativos. Estas variables se pueden utilizar en scripts y lógicas de control para implementar funcionalidades específicas y mejorar la interacción con el operador. \par 

En el contexto del proyecto, estas etiquetas permiten por ejemplo dar cumplimiento directo a los requisitos fundacionales de identificación y autenticación (\textit{FR1: IAC}) de la norma de ciberseguridad industrial IEC 62443-4-2:2019, registrando de forma única el operario activo (\texttt{@CurrentUser}) y la estación o cliente ligero de la Sala de Control Central (\texttt{@LocalMachineName}) desde el que se efectúa una orden. Asimismo, para garantizar un entorno distribuido robusto y tolerante a fallos de acuerdo con la arquitectura redundante implementada, el sistema monitoriza constantemente el rol lógico de las máquinas (Maestro o Reserva) y el estado de la sincronización de las bases de datos de Microsoft SQL Server mediante el bloque de variables de redundancia. \subsection{Variables de estructura}
Las variables de estructura son el nucleo de la solución, ya que permiten estructurar la información de forma lógica y coherente. Para el modelado de las variables de estructura se han seguido dos normas internacionales: La norma IEC 61400-25-2 para el modelado de los aerogeneradores y la norma IEC 61850 para el modelado de la subestación eléctrica. Estas normas establecen un marco común para la representación y organización de datos en sistemas eólicos y eléctricos.\par
\subsubsection{Modelo de información de aerogeneradores}
La norma IEC 61400-25 es una parte fundamental del conjunto de estándares internacionales IEC 61400, los cuales están dedicados a los sistemas de generación de energía eólica. En concreto, la serie "25" se centra exclusivamente en las comunicaciones para la monitorización y el control de parques eólicos. La parte "2" de esta serie se enfoca en la definición del \ac{IM}, los cuales definen el acceso al contenido de los datos. Este modelo de información permite la interoperabilidad entre los diferentes componentes de un sistema eólico, como turbinas, sistemas de control, sensores y sistemas de supervisión. Esta norma establece un marco para la integración y el funcionamiento coordinado de los sistemas eólicos a nivel global.\par

La norma establece cinco tipos de niveles en variables de estructura, cada uno con su propia función y propósito dentro del sistema de comunicación eólico:
\begin{itemize}
    \item \textbf{Nivel 1: Dispositivo Físico} Es el nivel más bajo de la jerarquía y representa los dispositivos físicos reales, como sensores, actuadores, controladores lógicos programables (PLC) y otros equipos de campo. 
    \item \textbf{Nivel 2: Dispositivo Lógico} Es un nivel de abstracción que agrupa nodos lógicos que comparten una misma ubicación física.
    \item \textbf{Nivel 3: Nodo Lógico} Es un nivel de abstracción que representa un componente mecanico real o una unidad funcional.
    \item \textbf{Nivel 4: Variable} Es un contenedor de datos (\textit{Tag}) que puede cambiar durante la ejecución del sistema.
    \item \textbf{Nivel 5: Atributo de Dato} Es una característica específica de un objeto que almacena información sobre ese objeto: Magnitud de la variable, calidad de la señal y tiempo de la última actualización.
\end{itemize}

\paragraph{Nivel 1 y 2: Estructura de Dispositivos Lógicos (LD) y Dispositivos Físicos (PD)}
El nivel 1 de estructura de dispositivos representa los dispositivos físicos reales a los que se accede a través de la red de comunicación. En el caso de un parque eólico, cada aerogenerador se modela como un Dispositivo Físico (PD) con un identificador único (Ej. \texttt{WTG1} para el aerogenerador 1). Este nivel refleja la realidad física de la planta, donde cada dispositivo tiene una ubicación geográfica y una función específica dentro del sistema de generación eólica. \par

El nivel 2 de estructura de dispositivos, por su parte, agrupa los nodos lógicos que comparten una misma ubicación física por ejemplo la góndola dentro de la turbina. En el caso de un aerogenerador que comparte sistemas pertenecientes a una misma turbina física, la fusión del Dispositivo Físico (Nivel 1) y Lógico (Nivel 2) resultaría en una representación más eficiente ya que todos sus subsistemas (\texttt{WROT}, \texttt{WGEN}, \texttt{WYAW}) comparten la misma ubicación geográfica y de red. Mantener una separación en el SCADA  resultaría redundante, por lo que su unificación bajo el prefijo único \texttt{WTGxx} optimiza el árbol de variables en \textit{SIMATIC WinCC V8.1}(Tabla \ref{tab:ejemplo_fusion_niveles}).

\paragraph{Nivel 3: Estructura de Nodos Lógicos (LN)}
Desde el punto de vista de la ingeniería electromecánica, un aerogenerador moderno es un sistema complejo constituido por múltiples subsistemas e componentes de hardware, tales como el rotor, la góndola, la torre, el sistema de orientación (*yaw*) y la estación meteorológica, entre otros (Figura \ref{fig:IEC_2}). Cada uno de estos componentes tiene una función específica en el proceso de captación y conversión de la energía eólica.\par

\begin{figure}[tb]
    \centering
    \includegraphics[width=\textwidth]{figuras/Nodos_1400.png}
    \caption{Descomposición de los componentes físicos de un aerogenerador y su correspondencia conceptual según la norma IEC 61400-25-2. Obtenido de \url{https://commons.wikimedia.org/wiki/File:IEC_61400-25-2_v1.jpg}}
    \label{fig:IEC_2}
\end{figure}

Con el fin de supervisar y automatizar el parque eólico, el estándar IEC 61400-25-2 proyecta esta estructura física hacia un modelo digital mediante una capa de abstracción basada en \ac{LN}. Un \ac{LN} no es un elemento material, sino una agrupación virtual de datos, variables y funciones de control que representan el comportamiento de un componente físico específico.\par 

Estos nodos se identifican de manera unívoca mediante siglas de tres o cuatro letras mayúsculas. El primer carácter determina la clase o categoría principal del objeto (por ejemplo, "W" para funciones específicas de la turbina), mientras que las letras sucesivas detallan la función o el subsistema concreto al que representan (por ejemplo, "ROT" para variables asociadas a la rotación del rotor, o "GEN" para los parámetros del generador). De este modo, toda la topología física del aerogenerador queda completamente digitalizada y organizada en áreas funcionales dentro del SCADA. A continuación, se detallan los nodos lógicos definidos para cada sección del aerogenerador:
\begin{itemize}
    \item \textbf{LLN0 (Logical Node Zero):} Nodo obligatorio que centraliza la información común del dispositivo lógico, tales como la configuración del sistema, versión del firmware y estados generales de operación.
    \item \textbf{LPHD (Physical Device Information):} Alberga las propiedades estructurales y físicas del hardware que soporta los nodos lógicos, incluyendo el número de serie, fabricante y el estado de salud del dispositivo (\textit{PhyHealth}).
    \item \textbf{WGEN (Generator):} Agrupa toda la información relativa a la conversión electromecánica y variables eléctricas de salida de la máquina.
    \item \textbf{WROT (Rotor):} Contiene las variables cinemáticas del conjunto de palas y buje, regulando parámetros esenciales como la velocidad angular del eje principal.
    \item \textbf{WYAW (Yaw System):} Gestiona los datos del sistema de orientación de la góndola respecto a la dirección predominante del viento.
    \item \textbf{WTOW (Tower):} Modela el comportamiento y las variables de la estructura de la torre.
    \item \textbf{WNAC (Nacelle):} Monitoriza las condiciones ambientales e instrumentación alojada en la góndola, tales como la temperatura interna, vibraciones y la velocidad/dirección del viento medidas por el anemómetro de techo.
    \item \textbf{WMET (Meteorological Station):} Registra las variables meteorológicas del entorno de la turbina, crucial para evaluar las curvas de potencia analíticas de la planta.
    \item \textbf{WTUR (Turbine):} Representa la turbina eólica completa, integrando las funciones de los componentes individuales.
\end{itemize}

\paragraph{Justificación de la Jerarquía de Direccionamiento y Multi-Subestación (6 Niveles vs. 5 Niveles)}
\label{sec:sub_jerarquia_6niveles}

Un análisis final de las variables creadas en el \ac{SCADA} revela que es necesaria la implementación de un nivel adicional de prefijo organizativo.\par

Tal como se recoge en el Anexo \ref{cap:Planos}, el proyecto no supervisa una infraestructura aislada, sino que consta de un total de 9 subestaciones eléctricas distribuidas a lo largo del complejo energético. Introducir las variables en el \textit{Tag Management} arrancando directamente desde el nivel  de celda (\texttt{Q01}) provocaría una colisión de direccionamiento en la base de datos de WinCC, ya que existirian celdas con la misma dirección. Una opción podria ser que cada subestación tuviera su propio numero de celda, sin embargo, por practicidad, se ha decidido implementar un nivel adicional de organización.\par

Este campo actúa como una raíz que encapsula los cinco niveles normativos de cada subestación independiente, estructurándose de la siguiente forma:

\begin{equation}
\underbrace{\text{SS\_xx}}_{\text{Subestación}} . \underbrace{\text{Q01}}_{\text{L1 \ac{PD}}} . \underbrace{\text{CTRL}}_{\text{L2 (\ac{LD})}} . \underbrace{\text{XCBR}}_{\text{L3 (\ac{LN})}} . \underbrace{\text{Pos}}_{\text{L4 (\ac{DO})}} . \underbrace{\text{stVal}}_{\text{L5 (\ac{DA})}}
\end{equation}.

\subsection{Variables externas}
Para la adquisición de datos el entorno de WinCC V8.1 ofrece drivers con una amplia variedad de protocolos y estandares de comunicación industrial desde comunicaciones propietarias como \texttt{S7} o \texttt{Profinet} a estandares abiertos como \ac{OPC UA}. Este proyecto principalmente estaba planteado para trabajar con el protocolo \texttt{Modbus TCP} pero se decanto por una solución basada en el estandar \ac{OPC UA} por su mayor flexibilidad, escalabilidad y seguridad en la transmisión de datos en tiempo real, además de su capacidad de integración con sistemas heterogéneos y su compatibilidad con la arquitectura orientada a objetos del SCADA.\par

\subsubsection{OPC UA}
\ac{OPC UA} es middleware de comunicación \ac{SOA} desarrollado por OPC Foundation y estandarizado por la norma IEC 62541\cite{iec62541_1}.\par

\ac{OPC UA} se basa en un modelo de información orientado a objetos, el cual define como se estructuran los datos dentro del sistema de comunicación industrial. Por otra parte, el modelo de comunicación de \ac{OPC UA} define los mecanismos de transferencia de los datos entre los diferentes componentes del sistema. El modelo de información y el modelo de comunicación son los dos pilares fundamentales que sustentan la arquitectura de \ac{OPC UA}.\par

\ac{OPC UA} se basa en un modelo Cliente-Servidor para gestionar servicios o ejecutar metodos como la conexión \ac{TCP} entre el servidor \ac{OPC UA} y el cliente \ac{SCADA} o gestionar la suscripción a tópicos. Sin embargo, la transfrerencia de datos se hace mediante publicación-suscripción (Pub/Sub) para optimizar la eficiencia en la transmisión en tiempo real(Figura \ref{fig:OPC_UA}).\par
\begin{figure}[tb]
    \centering
    \includegraphics[width=\textwidth]{figuras/OPCUA_metamodelo.png}
    \caption{Metamodelo de comunicación \ac{OPC UA}. Obtenido de Unified Automation GmbH. https://www.unified-automation.com/products/opc-ua/technology/opc-ua-metamodel.html}
    \label{fig:OPC_UA}
\end{figure}

\ac{OPC UA} incorpora características avanzadas de seguridad, como autenticación, autorización y cifrado, para proteger la integridad y confidencialidad de los datos transmitidos, además de especial enfoque en la capacidad de tiempo real y la independencia de la plataforma.\par

En el contexto del proyecto, KEPServerEX actúa como el servidor y broker \ac{OPC UA} que publica los datos del proceso simulados por NodeRed en los topicos, mientras que SIMATIC WinCC V8.1 se configura como cliente y consumidor \ac{OPC UA} quien consume la información a la que esta suscrita en tiempo real(Figura \ref{fig:OPC_UA_Kepserver}).\par

\section{Interfaz Gráfica}

La interfaz gráfica constituye el punto de interacción principal entre el operador y el sistema SCADA, por lo que su diseño debe priorizar la claridad, la rapidez de interpretación y la operabilidad en tiempo real. En esta sección se describe la propuesta de visualización desarrollada en WinCC, explicando la organización general de la interfaz y los criterios utilizados para su construcción.\par

Aunque en el diseño actual de interfaces industriales existe una fuerte tendencia hacia la adopción del estándar ISA-101 (High-Performance HMI) que aboga por el uso casi exclusivo de escalas de grises, gráficos vectoriales abstractos y la minimización radical del color, en el presente Trabajo Fin de Máster se ha decidido no aplicar este estándar de forma estricta por  razones técnicas y de alcance.\par

Por un lado, destaca el enfoque didáctico y demostrativo de esta prueba de concepto. Al abarcar desde la adquisición de datos de campo hasta la integración IIoT en la nube (AWS), una representación ilustrativa y a color facilita enormemente la comprensión inmediata del estado de los aerogeneradores durante la exposición del proyecto.\par

Por otro lado, se ha priorizado la optimización de recursos y la compatibilidad nativa empleando la \textit{Siemens Industry Graphic Library}. El aprovechamiento de estas paletas de color y controles predefinidos en WinCC V8.1 reduce los tiempos de diseño de la interfaz, permitiendo centrar los esfuerzos en arquitecturas más complejas como la comunicación \ac{OPC UA}.

Por último, dada la potencia del software de desarrollo SIMATIC WinCC V8.1, se ha optado por una representación que aproveche al maximo las capacidades gráficas del entorno, demostrando así la capacidad de la plataforma para crear interfaces visualmente atractivas y funcionales, sin comprometer la claridad ni la operatividad. En definitiva, a pesar de no sustentarse estrictamente en el estandar ISA-101, la propuesta se ha diseñado con criterios de usabilidad y eficacia.


\subsection{Jerarquía de pantallas}
En concreto, se presenta la jerarquía de pantallas definida para la navegación (desde vistas generales hasta vistas de detalle), la estructura de menús y accesos, la distribución de los elementos de supervisión y las pautas de representación de estados, alarmas y variables de proceso. Con ello se justifica cómo la arquitectura gráfica facilita tanto la monitorización continua como la operación segura del sistema.

Para garantizar la escalabilidad, el mantenimiento y la correcta integración de la navegación dentro del entorno de desarrollo SIMATIC WinCC V8.1, se ha implementado un sistema de nomenclatura estandarizado para todos los archivos gráficos de la interfaz (\texttt{.pdl}). Este sistema adopta una estructura jerárquica y funcional, resultando indispensable para la gestión eficiente de proyectos de automatización con un alto volumen de variables y pantallas.

El patrón de diseño definido para nombrar cada interfaz se fundamenta en tres bloques lógicos: \texttt{[Nivel]\_[Categoría]\_[Descripción\_Funcional]} (figura \ref{fig:jerarquia_pantallas}).
\begin{figure}[t]
    \centering
    \includegraphics[scale=0.45]{figuras/jerarquia.png}
    \caption{Ejemplo de jerarquía de pantallas en una interfaz gráfica de supervisión industrial.}
    \label{fig:jerarquia_pantallas}
\end{figure}
\subsubsection{Prefijo Numérico (Jerarquía y Orden de Navegación)}
El primer bloque numérico determina la profundidad de la pantalla dentro del árbol de navegación del SCADA, marcando el flujo operativo:
\begin{itemize}
    \item \textbf{00 (Nivel Raíz):} Reservado exclusivamente para el marco base de la aplicación y la pantalla de inicio principal (\texttt{00\_Main\_Interface}).Físicamente, la arquitectura gráfica se sustenta en esta plantilla maestra que actúa como Nivel 0.(figura \ref{fig:interfaz_grafica}).
    \item \textbf{1X (Áreas Principales):} Define las divisiones macroscópicas del parque eólico. Incluye la vista general de los aerogeneradores (\texttt{10}), la subestación eléctrica (\texttt{11}), la estación meteorológica (\texttt{12}) y los servicios de diagnóstico del sistema (\texttt{13}, \texttt{14}).
    \item \textbf{2X, 3X, 4X (Niveles de Detalle y Equipos):} Representan submenús y el desglose técnico de los activos. A medida que se incrementa el dígito inicial, aumenta el detalle de la información supervisada. Por ejemplo, los niveles 40, 41 y 42 se destinan a la monitorización de subsistemas de los aerogeneradores, como el generador o el rotor.
    \item \textbf{9X (Pop Ups y Faceplates):} Se reservan para pantallas de uso específico, como ventanas emergentes de alarmas, plantillas de variables (\textit{Faceplates}) o interfaces de configuración avanzada. Estas pantallas no forman parte del flujo de navegación principal, sino que se activan bajo demanda para tareas puntuales.
\end{itemize}

\subsubsection{Identificador de Categoría (Clasificación de Activos)}
El segundo bloque emplea acrónimos técnicos para identificar de forma unívoca la familia de sistemas a la que pertenece la interfaz:
\begin{itemize}
    \item \textbf{WTG (Wind Turbine Generator):} Pantallas dedicadas a la monitorización, control y operación directa de los aerogeneradores.
    \item \textbf{SUB (Substation):} Interfaz enfocada en la infraestructura eléctrica de evacuación, esquemas unifilares y transformadores.
    \item \textbf{MET (Meteorological):} Visualización de los datos climáticos adquiridos por la torre meteorológica.
    \item \textbf{SYS (System):} Funcionalidades globales del entorno SCADA, abarcando la gestión de alarmas y las pantallas de tendencias (\textit{Trends}).
\end{itemize}

\subsubsection{Descripción Funcional}
El tercer bloque explicita el propósito operativo de la pantalla, agilizando su localización por parte del programador o integrador:
\begin{itemize}
    \item \textbf{Overview:} Vista panorámica que agrega los indicadores clave (KPIs) y el estado general de un conjunto de activos.
    \item \textbf{Interface / Detail:} Panel de control principal o plantilla (\textit{Faceplate}) que detalla las variables operativas de un equipo.
    \item \textbf{SLD (Single Line Diagram):} Representación dinámica del esquema eléctrico unifilar.
    \item \textbf{Gen / Rotor / TRF:} Identificadores concretos de subsistemas electromecánicos (Generador, Rotor, Transformador).
\end{itemize}

\vspace{0.5cm}
La adopción de esta nomenclatura define la lógica de navegación requerida en la operación de plantas de generación. El operador inicia la supervisión en una vista global (\texttt{10\_WTG\_Overview}), focaliza la atención en una turbina particular (\texttt{30\_WTG\_Turbine\_Interface}) y profundiza en el diagnóstico de sus componentes mecánicos (\texttt{40\_WTG\_Gen}).\par 

Asimismo, esta estandarización resulta determinante para optimizar el rendimiento de la aplicación gráfica. Al mantener nombres predecibles y estructurados, se facilita el uso de \textit{scripts} y ventanas de imagen dinámicas (Figure \ref{fig:00_main_interface}), permitiendo que una única pantalla genérica actúe como interfaz para cualquier aerogenerador del parque mediante el paso de parámetros y prefijos de variables estructuradas.\par

\subsubsection{Plantillas dinámicas y reutilización gráfica (Prefijos de variable)}
\label{sec:tag_prefix}

En la monitorización de parques eólicos, se presenta una alta repetición de activos idénticos (decenas de aerogeneradores con la misma instrumentación). Desarrollar un archivo \texttt{.pdl} individual para cada turbina o clúster resultaría ineficiente, consumiendo recursos excesivos del sistema y dificultando el mantenimiento futuro del SCADA. 

Para resolver esta problemática, el diseño de las interfaces de detalle se ha fundamentado en el principio de reutilización gráfica, combinando el uso de las variables de estructura con la propiedad Prefijo de variable de las ventanas de imagen de SIMATIC WinCC.

La técnica consiste en diseñar una única pantalla genérica  donde los objetos gráficos (barras, campos de texto, indicadores de estado) se vinculan a las variables sin especificar la turbina a la que pertenecen (por ejemplo, apuntando a la variable \texttt{.WROT.RotAzi.mag} en lugar de \texttt{WTG1.WROT.RotAzi.mag}).

Durante la operación en el entorno de ejecución, cuando el operador selecciona un aerogenerador específico desde la vista general, se ejecuta un \textit{script} que realiza dos acciones simultáneas sobre la ventana de imagen central:
\begin{enumerate}
    \item Carga el archivo \texttt{.pdl} de la plantilla genérica.
    \item Asigna el nombre de la turbina seleccionada (ej. \texttt{WTG1}) a la propiedad de prefijo de variable de la ventana.
\end{enumerate}

Automáticamente, WinCC concatena este prefijo con el sufijo interno de los objetos gráficos, dinamizando la pantalla completa con los datos reales del equipo seleccionado en cuestión de milisegundos (ver Figura \ref{fig:plantillas_dinamizadas}). Esta filosofía se ha aplicado intensivamente en los niveles del proyecto.

\section{Gestión de alarmas y eventos}
\label{sec:alarmas}

La gestión de alarmas y eventos es el componente crítico del sistema SCADA encargado de garantizar la seguridad operativa y la conciencia situacional del operador. Su función principal no es solo la detección de anomalías, sino la jerarquización de las mismas para evitar la saturación de información durante incidentes críticos. 

En este proyecto, la supervisión de eventos se integra de forma transversal mediante la ventanta de imagen inferior persistente en la pantalla \texttt{00\_Main\_Interface}. Este diseño asegura que, independientemente del nivel de navegación en el que se encuentre el usuario, las notificaciones de mayor prioridad sean visibles de forma inmediata. La implementación se ha realizado empleando el objeto \textit{WinCC Alarm Control}, configurando ciclos de refresco en tiempo real y almacenamiento en una base de datos SQL histórica para auditorías posteriores.

\subsection{Criterios de señalización: Norma IEC 60073}

Para garantizar una interpretación unívoca de los estados del proceso, se ha seguido la normativa internacional IEC 60073(\textit{Principios básicos y de seguridad para las interfaces hombre-máquina, el marcado y la identificación}) \cite{iec60073}. Esta norma establece una codificación cromática estricta basada en el significado de la seguridad y el estado del equipo:

\begin{itemize}
    \item \textbf{Rojo:} Indica una situación de peligro, emergencia o actuación inmediata.
    \item \textbf{Amarillo/Ámbar:} Señala una condición anormal o un cambio crítico de estado que requiere precaución o supervisión.
    \item \textbf{Verde:} Representa una condición normal de funcionamiento o seguridad.
    \item \textbf{Azul:} Indica una condición de carácter obligatorio o una intervención específica del operador.
    \item \textbf{Blanco/Gris:} Utilizado para estados neutros o falta de información específica de seguridad.
\end{itemize}

\subsection{Configuración e implementación del sistema de avisos en SIMATIC WinCC}
\label{sec:configuracion_alarmas_wincc}

La materialización de la gestión de alarmas en el entorno \textit{Runtime} se estructura mediante el subsistema \textit{WinCC Alarm Logging}. Para mantener la coherencia con el modelo de información estandarizado (Capitulo \ref{cap:modelo_informacion}), todos los eventos discretos del aerogenerador se han concentrado en el Nodo Lógico de alarmas, denominado \texttt{WALM} (ver Anexo \ref{cap:Tablas_Variab}).

La categorización de estos eventos determina su gravedad y a consecuencia su color en la interfaz gráfica, regulando la interacción mediante el bloqueo, acuse y reconocimiento.

\subsubsection{Categorización funcional}

Las señales de error, configuradas en WinCC como variables de tipo \textit{Single Point Status (\ac{SPS})}, se clasifican en tres clases fundamentales de avisos:

\begin{itemize}
    \item \textbf{Alarmas de Emergencia (Críticas):} Eventos que implican un riesgo inminente para la integridad de la máquina o de la red, provocando la parada inmediata de la turbina. 
    \item \textbf{Alarmas de Fallo (Operativas):} Indican averías de componentes mecánicos o eléctricos que interrumpen la producción, pero mantienen el sistema en un estado controlado. 
    \item \textbf{Advertencias (Avisos):} Notifican condiciones ambientales o desviaciones técnicas que requieren atención preventiva sin forzar una parada inmediata.
\end{itemize}

\section{Integración de IIoT en la solución}

En el contexto actual de la Industria 4.0, la monitorización y supervisión de infraestructuras energéticas distribuidas, como es el caso de un parque eólico, se demandan soluciones que vayan más allá de los sistemas \ac{SCADA} tradicionales aislados en planta. La convergencia entre \ac{IT} y \ac{OT} permite habilitar el ecosistema del \ac{IIoT}, facilitando el análisis de datos a gran escala, la accesibilidad remota y la escalabilidad del sistema.\par

Este capítulo describe el despliegue e integración de una arquitectura en la nube orientada a la adquisición y visualización de datos  provenientes de los aerogeneradores. Mientras que el sistema local (basado en SIMATIC WinCC V8.1) se encarga de la adquisición en tiempo real (Edge) y la monitorización a nivel de campo mediante protocolos industriales como \ac{OPC UA}, la arquitectura \ac{IIoT} tiene como objetivo exportar estos datos hacia la nube para su almacenamiento histórico y explotación visual.\par

Para lograr un sistema robusto y replicable, se ha diseñado una arquitectura basada en microservicios contenerizados, apoyada en las siguientes capas lógicas (figura \ref{fig:IIoT}):

\begin{itemize}
    \item \textbf{Capa de Adquisición y Emisión (Edge)}: Representada por el módulo Cloud Connect del \ac{SCADA} WinCC, encargado de empaquetar las variables de proceso (velocidad del viento, potencia generada, estados, etc.) y publicarlas mediante el protocolo de mensajería ligero \ac{MQTT}.
    \item \textbf{Capa de Infraestructura (Cloud)}: Soportada por una instancia virtua \ac{EC2} alojada en \ac{AWS}, que proporciona el entorno computacional necesario con alta disponibilidad. El despliegue de los servicios se ha orquestado íntegramente mediante Docker, garantizando el aislamiento de los procesos y facilitando futuras migraciones.
    \item \textbf{Capa de Ingesta y Procesamiento Intermedio}: Implementada con Mosquitto y Node-RED. Mosquitto actua como el broker \ac{MQTT} mientras que Node-RED consume los datos y proporciona un entorno de flujo de datos visual para filtrar y transformar los paquetes \ac{JSON} recibidos antes de su inserción en la base de datos.
    \item \textbf{Capa de Almacenamiento}: Soportada por InfluxDB, una base de datos de series temporales optimizada para  el almacenamiento eficiente de datos con marcas de tiempo.
    \item \textbf{Capa de Visualización}: Desarrollada sobre Grafana, que consulta directamente a la base de datos para construir dashboards interactivos, permitiendo la monitorización del rendimiento del parque eólico desde cualquier ubicación con acceso a internet.
\end{itemize}

\subsubsection{Orquestación de servicios mediante Docker}
\label{sec:despliegue_iiot_docker}

Para garantizar la portabilidad, el aislamiento y la escalabilidad de la solución IIoT orientada al parque eólico, se ha optado por contenedorizar cada componente de la arquitectura (Node-RED, Mosquitto, InfluxDB y Grafana) utilizando la plataforma Docker. Docker es un motor de virtualización a nivel de sistema operativo que permite empaquetar una aplicación y todas sus dependencias binarias en un artefacto único e independiente.\par 

A diferencia de las máquinas virtuales tradicionales, que requieren la emulación completa de un sistema operativo invitado sobre un hipervisor, Docker opera mediante el concepto de contenedor. Un contenedor es un proceso aislado y ligero que comparte el mismo núcleo (\textit{kernel}) del sistema operativo host, abstrayendo únicamente las capas de software superiores (Figura \ref{fig:docker_architecture}). De esta forma, las configuraciones específicas de los diferentes entornos donde se puedan desplegar los servicios (desarrollo, pruebas o producción industrial) se gestionan de forma consistente, replicable y con un consumo de recursos computacionales mínimo.

\section{Ciberseguridad IEC 62443-4-2:2019}
Esta sección es una directriz de seguridad válida para SIMATIC HMI WinCC V8.1 que describe sus funciones de ciberseguridad. Sigue la norma IEC 62443-4-2:2019 de ciberseguridad para sistemas de automatización y control industrial. La norma IEC 62443 define cuatro \ac{SL} distintos, basados en los recursos, las habilidades y la motivación del posible atacante \cite{iec62443-4-2}:

\begin{itemize}
    \item \textbf{\ac{SL} 1:} Protección contra vulneraciones casuales/accidentales:
    Protección contra el acceso no autorizado por parte de alguien sin intención maliciosa.
    \item \textbf{\ac{SL} 2:} Protección contra vulneraciones intencionadas con medios sencillos:
    Protección contra ataques genéricos y de bajos recursos.
    \item \textbf{\ac{SL} 3:} Protección contra ataques sofisticados:
    Protección contra atacantes (hackers) con recursos moderados, motivación moderada y conocimientos específicos sobre Sistemas de Automatización y Control Industrial (IACS, por sus siglas en inglés).
    \item \textbf{\ac{SL} 4:} Protección contra ataques altamente sofisticados / de Estados-nación:
    Protección contra atacantes con amplios recursos, alta motivación y profunda experiencia en IACS.
\end{itemize}

Los \ac{FR} son los requisitos de seguridad que deben cumplirse para alcanzar el SL específico. Estos requisitos se dividen en siete categorías principales, cada una con sub-requisitos específicos. Siemens ha evaluado su producto SIMATIC HMI WinCC V8.1 en función de estos requisitos y ha determinado que siguiendo los pasos recomendados en la documentación oficial proporcionada por ellos mismos, se puede alcanzar un nivel de seguridad \ac{SL} 2, con capacidades parciales para \ac{SL} 3 mediante la adecuada configuración de la segmentación de red y los mecanismos de detección \cite{siemens_wincc_security}.\par\indent

Para dar cumplimiento específico al requisito fundamental \textit{FR 1 (Identification and Authentication Control)} bajo las directrices del nivel de seguridad \ac{SL} 2, se prohíbe explícitamente el empleo de cuentas genéricas o compartidas en la planta. El sistema debe garantizar la identificación única de cada usuario.

Adicionalmente, se han implementado políticas de gestión de autenticación que incluyen caducidad de contraseñas y mecanismos de bloqueo de cuentas para prevenir accesos no autorizados. Las contraseñas deben tener una longitud mínima de 8 caracteres y combinar obligatoriamente letras mayúsculas y minúsculas, números y caracteres especiales. Además, ante cinco intentos de acceso fallidos consecutivos se bloquea automáticamente el identificador de usuario para mitigar ataques por fuerza bruta o diccionario. Por último, se configura el cierre automático de sesión en los terminales HMI tras 15 minutos de inactividad, protegiendo así las estaciones desatendidas en la sala de control.
