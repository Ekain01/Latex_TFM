\section{Introducción al entorno de SIMATIC WinCC V8.1}
SIMATIC WinCC V8.1 es un sistema \ac{SCADA}, basado en los sistemas operativos Microsoft Windows, desarrollado por Siemens AG. Está diseñado para la visualización, control y operación de procesos, flujos de producción, máquinas y plantas críticas en diversos sectores industriales.

\par \indent WinCC actúa como una \ac{HMI} de alto nivel. A diferencia de los paneles de operador básicos ubicados a pie de máquina, un \ac{HMI} de alto nivel como WinCC proporciona un entorno gráfico avanzado que permite a los operadores del centro de control manejar, monitorizar y entender de forma intuitiva los procesos dinámicos que se ejecutan en la planta sin tener que estar en la ubicación de la máquina. La comunicación entre el entorno de supervisión WinCC y la maquinaria física se efectúa de manera bidireccional a través de sistemas de automatización, empleando protocolos de comunicación industrial.(\ref{fig:wincc_interface})

\par \indent Por otro lado, WinCC es plataforma altamente programable. Integra un motor de ejecución que permite la creación de lógicas complejas y automatización de tareas de supervisión utilizando interfaces de aplicación  como \ac{ANSI-C} y VBScript (Visual Basic).Por otro lado, WinCC V8.1 utiliza de forma nativa el motor de bases de datos Microsoft SQL Server. Este gestor de bases de datos opera en segundo plano y se encarga del archivado histórico continuo de las variables del proceso y del registro de eventos y fallos.

\subsection{Sistemas Monopuesto (\textit{Single User Systems})}
En la arquitectura de sistema monopuesto un único equipo físico o estación se encarga de realizar todas las tareas del \ac{SCADA}. El puesto actúa tanto como servidor como cliente: El equipo se encarga de realizar las comunicaciones con la parte operativa (mediante los protocolos de comunicación como Profinet, ModBus u \ac{OPC UA} etc.) de ejecutar el software \ac{RT} y almacenar los históricos y alarmas en la base de datos SQL a la vez que muestra la interfaz gráfica al operario en los monitores. \par
Para este tipo de sistemas Siemens ofrece dos arquitecturas posibles: La primera se trata de una estación de configuración y \ac{RT} con un único PC (Figura \ref{fig:monopuesto_simple}). La segunda configuración 


es una configuración redundante de la primera donde la estación redundante solo sería una estación \ac{RT} (Figura \ref{fig:monopuesto_redundante}).

\subsection{Sistemas Multipuesto (\textit{Multy User Systems})}
Un sistema multipuesto es una arquitectura distribuida diseñada para separar las tareas de procesamiento de datos de las tareas de visualización. A diferencia de un sistema monopuesto (donde una única estación de operación realiza todas las tareas), en un sistema multipuesto la carga de trabajo se divide entre uno o varios Servidores centrales (configuradas como \textit{Multi-User Project}) y múltiples estaciones Cliente (configuradas como \textit{Client Project}). Este enfoque permite una variedad grande de arquitecturas y configuraciones de proyecto, adaptándose a las necesidades específicas de cada instalación, siendo el foco de implementación final. (Figura \ref{fig:multipuesto_distribuido}) \par 

\section{Virtualización mediante \ac{SIVaaS}}
Las arquitecturas de sistemas WinCC  pueden ser implementadas de forma física o virtualizada.
La virtualización es un método por el que los recursos físicos de un equipo se pueden dividir en diversos entornos virtuales. Para ello se utiliza un hipervisor que se encarga de ejecutar múltiples \ac{VM} aisladas, compartiendo recursos como \ac{CPU}, memoria y almacenamiento. Este aislamiento evita los conflictos debidos a las dependencias de software y proporciona la capacidad de iniciar y detener las máquinas virtuales de forma independiente. Solo la máquina virtual en cuestión se ve afectada por un fallo (o caída del sistema). Existen dos tipos de hipervisores: Alojados y Metal-Desnudos.
\subsection{Hypervisor alojado  (\textit{Hosted Hypervisor})}
Este tipo se instala como una aplicación de software más dentro de un sistema operativo convencional que ya está funcionando en el equipo físico (conocido como \ac{OS} Anfitrión o Host \ac{OS}).\par

Este tipo de hipervisor genera una dependencia del \ac{OS} base ya que el hipervisor debe pedir constantemente permiso para acceder a los recursos del sistema lo cual genera un cuello de botella disminuyendo el rendimiento. \par

Por otro lado, esta dependencia del \ac{OS} base es una vulnerabilidad en sí misma para la seguridad y ciberseguridad del sistema, ya que un malfuncionamiento o un virus podrían contagiarse a cada una de las instancias de maquinas virtuales que corren encima.

\subsection{Hypervisor metal desnudo (\textit{Bare-Metal Hypervisor})}

Se le denomina \textit{Bare-Metal} (metal desnudo) porque se instala y ejecuta directamente sobre el hardware físico del servidor. No hay ningún sistema operativo tradicional (como Windows Server o Linux) actuando como intermediario. El hipervisor en sí mismo es un sistema operativo altamente especializado y minimalista diseñado con un único propósito: gestionar y asignar recursos físicos a las máquinas virtuales. Al no tener que atravesar ningún sistema operativo anfitrión, las maquinas virtuales tienen acceso más directo a los recursos del sistema reduciendo la latencia y sobrecarga. \par

Al contrario que un hipervisor alojado, es seguro ante vulnerabilidades, debido a que que tiene una superficie de ataque reducida. Además, las máquinas virtuales están aisladas por lo que si una maquina virtual se ve comprometida por un malware el resto de maquinas permanecerán seguras.\par

\subsection{Virtualización SIMATIC como Servicio (\ac{SIVaaS})}
Siemens ofrece una solución de virtualización completa para sistemas de control basada en hipervisores bare-metal llamada Virtualización SIMATIC como Servicio (\ac{SIVaaS}). \ac{SIVaaS} es una solución de virtualización completa para sistemas de control basados en PC. El hardware, el software, la configuración y los servicios de ciclo de vida están totalmente integrados, abarcando desde la configuración del host y el hipervisor hasta las máquinas virtuales, los sistemas operativos y la instalación de software llave en mano. \par

\textbf{ESPECIFICAR EL CAJON QUE SE UTILIZARA (PLANOS DE LOS SERVIDORES)}
\section{\ac{IM} de la solución SCADA}
\label{cap:modelo_informacion}
El modelo de información de un sistema es la representación estructurada de los datos y la información que maneja ese sistema. Con ello se define la forma en que se organizan, almacenan y acceden a los datoa dentro del sistema. El entorno de SIMATIC WinCC V8.1 permite la creación de variables que se utilizan para representar los datos del proceso, las alarmas, los eventos y otros aspectos operativos del sistema SCADA. Estas variables pueden ser internas, externas o de estructura, dependiendo de su función y origen dentro del sistema. \par

 Las variables internas son aquellas que se crean y gestionan dentro del entorno de WinCC, mientras que las variables externas son aquellas que se adquieren a través de la comunicación con los dispositivos físicos o sistemas externos. Las variables de estructura, por otro lado,son variables orientadas a objetos las cuales sirven para estructurar la información. Este tipo de variables son muy útiles para organizar y replicar la información de forma lógica y coherente.\par
\subsection{Variables internas}
El entorno de SIMATIC WinCC V8.1 ofrece una lista de variables internas predeterminadas que se utilizan para diversas funciones dentro del sistema SCADA. Estas variables internas son fundamentales para la operación del sistema, ya que proporcionan información crítica sobre el estado del sistema, el usuario actual, la máquina local, el idioma activo, la hora local, entre otros aspectos operativos. Estas variables se pueden utilizar en scripts y lógicas de control para implementar funcionalidades específicas y mejorar la interacción con el operador. \par 

En el contexto del proyecto, estas etiquetas permiten por ejemplo dar cumplimiento directo a los requisitos fundacionales de identificación y autenticación (\textit{FR1: IAC}) de la norma de ciberseguridad industrial \textbf{IEC 62443-4-2:2019}, registrando de forma única el operario activo (\texttt{@CurrentUser}) y la estación o cliente ligero de la Sala de Control Central (\texttt{@LocalMachineName}) desde el que se efectúa una orden. Asimismo, para garantizar un entorno distribuido robusto y tolerante a fallos de acuerdo con la arquitectura redundante implementada, el sistema monitoriza constantemente el rol lógico de las máquinas (Maestro o Reserva) y el estado de la sincronización de las bases de datos de Microsoft SQL Server mediante el bloque de variables de redundancia. Las variables internas mas destacadas se detallan en la Tabla~\ref{tab:variables_sistema_wincc}.

\subsection{Variables de estructura}
Las variables de estructura son el nucleo de la solución, ya que permiten estructurar la información de forma lógica y coherente. Para el modelado de las variables de estructura se han seguido dos normas internacionales: La norma IEC 61400-25-2 para el modelado de los aerogeneradores y la norma IEC 61850 para el modelado de la subestación eléctrica. Estas normas establecen un marco común para la representación y organización de datos en sistemas eólicos y eléctricos.\par
\subsubsection{Modelo de información de aerogeneradores}
La norma \textbf{IEC 61400-25} es una parte fundamental del conjunto de estándares internacionales IEC 61400, los cuales están dedicados a los sistemas de generación de energía eólica. En concreto, la serie "25" se centra exclusivamente en las comunicaciones para la monitorización y el control de parques eólicos. La parte "2" de esta serie se enfoca en la definición de los modelos de información (IM), los cuales definen el acceso al contenido de los datos. Este modelo de información permite la interoperabilidad entre los diferentes componentes de un sistema eólico, como turbinas, sistemas de control, sensores y sistemas de supervisión. Esta norma establece un marco para la integración y el funcionamiento coordinado de los sistemas eólicos a nivel global.\par

La norma establece cinco tipos de niveles en variables de estructura, cada uno con su propia función y propósito dentro del sistema de comunicación eólico:
\begin{itemize}
    \item \textbf{Nivel 1: Dispositivo Físico} Es el nivel más bajo de la jerarquía y representa los dispositivos físicos reales, como sensores, actuadores, controladores lógicos programables (PLC) y otros equipos de campo. 
    \item \textbf{Nivel 2: Dispositivo Lógico} Es un nivel de abstracción que agrupa nodos lógicos que comparten una misma ubicación física.
    \item \textbf{Nivel 3: Nodo Lógico} Es un nivel de abstracción que representa un componente mecanico real o una unidad funcional.
    \item \textbf{Nivel 4: Variable} Es un contenedor de datos (\textit{Tag}) que puede cambiar durante la ejecución del sistema.
    \item \textbf{Nivel 5: Atributo de Dato} Es una característica específica de un objeto que almacena información sobre ese objeto: Magnitud de la variable, calidad de la señal y tiempo de la última actualización.
\end{itemize}

\paragraph{Nivel 1 y 2: Estructura de Dispositivos Lógicos (LD) y Dispositivos Físicos (PD)}
El nivel 1 de estructura de dispositivos representa los dispositivos físicos reales a los que se accede a través de la red de comunicación. En el caso de un parque eólico, cada aerogenerador se modela como un Dispositivo Físico (PD) con un identificador único (Ej. \texttt{WTG1} para el aerogenerador 1). Este nivel refleja la realidad física de la planta, donde cada dispositivo tiene una ubicación geográfica y una función específica dentro del sistema de generación eólica. \par

El nivel 2 de estructura de dispositivos, por su parte, agrupa los nodos lógicos que comparten una misma ubicación física por ejemplo la góndola dentro de la turbina. En el caso de un aerogenerador que comparte sistemas pertenecientes a una misma turbina física, la fusión del Dispositivo Físico (Nivel 1) y Lógico (Nivel 2) se podría justificar por su centralización física y funcional ya que todos sus subsistemas (\texttt{WROT}, \texttt{WGEN}, \texttt{WYAW}) comparten la misma ubicación geográfica y de red. Mantener una separación en el SCADA  resultaría redundante, por lo que su unificación bajo el prefijo único \texttt{WTGxx} optimiza el árbol de variables en \textit{SIMATIC WinCC V8.1}(Tabla \ref{tab:ejemplo_fusion_niveles}).

\paragraph{Nivel 3: Estructura de Nodos Lógicos (LN)}
El nivel 3 representa una agrupación de objetos que comparten una misma funcionalidad lógica. Se identifican mediante siglas de tres o cuatro letras mayúsculas siendo la primera letra la que indica el tipo de objeto (Por ejemplo W para la turbina) seguido por letras que indican la función o característica específica del objeto (por ejemplo, ROT para objetos relacionados con la rotación de la turbina). Los nodos lógicos se agrupan por subsistemas, cada uno de ellos representando un área funcional específica dentro del parque eólico. Se han definido los siguientes nodos lógicos para cada aerogenerador:
\begin{itemize}
    \item \textbf{LLN0 (Logical Node Zero):} Nodo obligatorio que centraliza la información común del dispositivo lógico, tales como la configuración del sistema, versión del firmware y estados generales de operación.
    \item \textbf{LPHD (Physical Device Information):} Alberga las propiedades estructurales y físicas del hardware que soporta los nodos lógicos, incluyendo el número de serie, fabricante y el estado de salud del dispositivo (\textit{PhyHealth}).
    \item \textbf{WGEN (Generator):} Agrupa toda la información relativa a la conversión electromecánica y variables eléctricas de salida de la máquina.
    \item \textbf{WROT (Rotor):} Contiene las variables cinemáticas del conjunto de palas y buje, regulando parámetros esenciales como la velocidad angular del eje principal.
    \item \textbf{WYAW (Yaw System):} Gestiona los datos del sistema de orientación de la góndola respecto a la dirección predominante del viento.
    \item \textbf{WTRF (Transformer):} Supervisa el estado de los activos de transformación eléctrica de media tensión integrados en la torre o peana.
    \item \textbf{WTOW (Tower):} Modela el comportamiento y las variables de la estructura de la torre.
    \item \textbf{WNAC (Nacelle):} Monitoriza las condiciones ambientales e instrumentación alojada en la góndola, tales como la temperatura interna, vibraciones y la velocidad/dirección del viento medidas por el anemómetro de techo.
    \item \textbf{WMET (Meteorological Station):} Registra las variables meteorológicas del entorno de la turbina, crucial para evaluar las curvas de potencia analíticas de la planta.
    \item \textbf{WTUR (Turbine):} Representa la turbina eólica completa, integrando las funciones de los componentes individuales.
\end{itemize}

\paragraph{Justificación de la Jerarquía de Direccionamiento y Multi-Subestación (6 Niveles vs. 5 Niveles)}
\label{sec:sub_jerarquia_6niveles}

Un análisis de la sintaxis final de los \textit{tags} creados en el SCADA revela que es necesaria la implementación de un nivel adicional de prefijo organizativo.\par

Tal como se recoge en el Anexo \ref{cap:Planos}, el proyecto no supervisa una infraestructura aislada, sino que consta de un total de \textbf{9 subestaciones eléctricas} distribuidas a lo largo del complejo energético. Introducir las variables en el \textit{Tag Management} arrancando directamente desde el nivel  de celda (\texttt{Q01}) provocaría una colisión masiva de direccionamiento en la base de datos de WinCC, ya que existirian celdas con la misma dirección.\par

Por este motivo, se ha implementado un Nivel 0 de subestación en adición a la norma. Este campo actúa como una raíz que encapsula los cinco niveles normativos de cada subestación independiente, estructurándose de la siguiente forma:

\begin{equation}
\underbrace{\text{SS\_xx}}_{\text{Prefijo SCADA (Planta)}} . \underbrace{\text{Q01}}_{\text{L1 (IED)}} . \underbrace{\text{CTRL}}_{\text{L2 (LD)}} . \underbrace{\text{XCBR1}}_{\text{L3 (LN)}} . \underbrace{\text{Pos}}_{\text{L4 (DO)}} . \underbrace{\text{stVal}}_{\text{L5 (DA)}}
\end{equation}.

\subsection{Variables externas}
Para la adquisición de datos el entorno de WinCC V8.1 ofrece drivers con una amplia variedad de protocolos y estandares de comunicación industrial como  Open Platform Communication Unified Architecture (\ac{OPC UA}). 

\subsubsection{OPC UA}
\ac{OPC UA} es middleware de comunicación orientado a servicios (SOA) desarrollado por OPC Foundation y estandarizado por la norma \textbf{IEC 62541}\cite{iec62541_1}.\par

\ac{OPC UA} se basa en un modelo de información orientado a objetos, el cual define como se estructuran los datos dentro del sistema de comunicación industrial. Por otra parte, el modelo de comunicación de \ac{OPC UA} define los mecanismos de transferencia de los datos entre los diferentes componentes del sistema. El modelo de información y el modelo de comunicación son los dos pilares fundamentales que sustentan la arquitectura de \ac{OPC UA}.\par

\ac{OPC UA} se basa en un modelo Cliente-Servidor para establecer la conexion TCP entre el servidor \ac{OPC UA} y el cliente SCADA. Sin embargo, una vez establecida la sesión, la transfrerencia se hace mediante publicación-suscripción (Pub/Sub) para optimizar la eficiencia en la transmisión de datos en tiempo real(Figura \ref{fig:OPC_UA}).\par
\begin{figure}[tb]
    \centering
    \includegraphics[width=\textwidth]{figuras/OPCUA_metamodelo.png}

\ac{OPC UA} incorpora características avanzadas de seguridad, como autenticación, autorización y cifrado, para proteger la integridad y confidencialidad de los datos transmitidos, además de especial enfoque en la capacidad de tiempo real y la independencia de la plataforma.\par

En el contexto del proyecto, KEPServerEX actúa como el servidor \ac{OPC UA} que publica los datos del proceso simulador por NodeRed en los topicos, mientras que SIMATIC WinCC V8.1 se configura como cliente \ac{OPC UA} quien consume la información a la que esta suscrita en tiempo real(Figura \ref{fig:OPC_UA_Kepserver}).\par

\subsubsection{Integración de Variables Externas en el \textit{Tag Management} mediante Direccionamiento \ac{OPC UA}}
\label{sec:integracion_tag_management_opcua}
Las conexiones de las variables de estructura con variables externas de \ac{OPC UA} se ha realizado en el servicio de administración de variables. Para que el SCADA pueda consumir la información de los aerogeneradores y subestaciones, se ha configurado la jerarquía de comunicaciones y establecido el mapeo  de las direcciones de red para cada variable externa.\par

\section{Modelo de Intercambio de Información (IEM) de la solución SCADA}
\label{subsec:modelo_intercambio_solucion}

Mientras que el modelo de información estático determina la semántica y la estructura orientada a objetos de los datos, el modelo de intercambio de información define de manera abstracta las reglas, servicios y verbos mediante los cuales la MCR  interactúa con los datos del parque eólico. En esta solución, el modelo de intercambio abstracto heredado de la norma IEC 61400-25-3 (Figura \ref{fig:IEM}) se ha materializado mediante un mapeo tecnológico directo sobre el estándar \textit{OPC Unified Architecture} (\ac{OPC UA}) coordinado entre \textit{KepServerEX} como servidor de comunicaciones y \textit{SIMATIC WinCC V8.1} como plataforma SCADA.
\begin{figure}[tb]
    \centering
    \includegraphics[width=\textwidth]{figuras/IEM.png}
    \caption{Modelo de Intercambio de Información (IEM) de la solución SCADA basado en \ac{OPC UA}. Obtenido de \url{https://webstore.iec.ch/en/iec_catalog/product/preview/?id=L3B1Yi9wZGYvcHJldmlldy9pbmZvX2llYzYxNDAwLTI1LTN7ZWQyLjAuUkxWfWVuLnBkZg==}}
    \label{fig:IEM}
\end{figure}
La implementación práctica de estos servicios de intercambio se articula en base a cuatro pilares operativos dentro de la arquitectura del proyecto:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Recuperación de Datos por Eventos (\textit{Exception Reporting}):} Para evitar la saturación computacional del \textit{Runtime} que provoca el \textit{polling} cíclico convencional de 120 turbinas concurrentes, el intercambio de datos en tiempo real se delega en el mecanismo de \textit{Monitored Items} de \ac{OPC UA}. Bajo este esquema, el canal de comunicación permanece en silencio y el controlador de campo solo notifica de forma espontánea una actualización hacia WinCC si la variable física (v.g., la velocidad del rotor \texttt{WROT.RotSpd}) rebasa una banda muerta (\textit{deadband}) preconfigurada en el servidor.
    
    \item \textbf{Notificación Asíncrona de Alarmas y Condiciones:} La gestión de estados críticos y disparos de los interruptores de la subestación (nodos \texttt{XCBR} bajo IEC 61850) implementa los bloques de control de informes (\textit{Report Control Blocks}). Al producirse una contingencia, el servidor inyecta una ráfaga de eventos asíncronos embebidos con marcas de tiempo de origen (\textit{Source Timestamp}). El módulo \textit{WinCC Alarm Logging} absorbe estas tramas sin bloquear los hilos gráficos, garantizando la trazabilidad cronológica exacta del fallo.
    
    \item \textbf{Persistencia y Registro Histórico (\textit{Logging}):} El servicio de almacenamiento histórico se ha implementado en un régimen de doble redundancia operativa. A nivel local en la MCR, WinCC realiza la captura de variables mediante \textit{Tag Logging} sobre una base de datos Microsoft SQL integrada para alimentar las gráficas de tendencia instantáneas del operador. De manera paralela y para salvaguardar el rendimiento local, un cliente de intercambio secundario instrumentado en \textit{Node-RED} captura los flujos industriales y los transfiere mediante la red IT hacia la base de datos de series temporales \textit{InfluxDB} en la nube de AWS, desacoplando la analítica a largo plazo de la operación en tiempo real.
    
\end{enumerate}

\section{Interfaz Gráfica}

La interfaz gráfica constituye el punto de interacción principal entre el operador y el sistema SCADA, por lo que su diseño debe priorizar la claridad, la rapidez de interpretación y la operabilidad en tiempo real. En esta sección se describe la propuesta de visualización desarrollada en WinCC, explicando la organización general de la interfaz y los criterios utilizados para su construcción.\par

Aunque en el diseño actual de interfaces industriales existe una fuerte tendencia hacia la adopción del estándar ISA-101 (High-Performance HMI) —que aboga por el uso casi exclusivo de escalas de grises, gráficos vectoriales abstractos y la minimización radical del color—, en el presente Trabajo Fin de Máster se ha decidido no aplicar este estándar de forma estricta por  razones técnicas y de alcance.\par

Por un lado, destaca el enfoque didáctico y demostrativo de esta prueba de concepto. Al abarcar desde la adquisición de datos de campo hasta la integración IIoT en la nube (AWS), una representación ilustrativa y a color facilita enormemente la comprensión inmediata del estado de los aerogeneradores durante la exposición del proyecto.\par

Por otro lado, se ha priorizado la optimización de recursos y la compatibilidad nativa empleando la \textit{Siemens Industry Graphic Library}. El aprovechamiento de estas paletas de color y controles predefinidos en WinCC V8.1 reduce los tiempos de diseño de la interfaz, permitiendo centrar los esfuerzos en arquitecturas más complejas como la comunicación \ac{OPC UA}.

Por último, dada la potencia del software de desarrollo SIMATIC WinCC V8.1, se ha optado por una representación que aproveche al maximo las capacidades gráficas del entorno, demostrando así la capacidad de la plataforma para crear interfaces visualmente atractivas y funcionales, sin comprometer la claridad ni la operatividad. En definitiva, a pesar de no sustentarse estrictamente en el estandar ISA-101, la propuesta se ha diseñado con criterios de usabilidad y eficacia.


\subsection{Jerarquía de pantallas}
En concreto, se presenta la jerarquía de pantallas definida para la navegación (desde vistas generales hasta vistas de detalle), la estructura de menús y accesos, la distribución de los elementos de supervisión y las pautas de representación de estados, alarmas y variables de proceso. Con ello se justifica cómo la arquitectura gráfica facilita tanto la monitorización continua como la operación segura del sistema.

Para garantizar la escalabilidad, el mantenimiento y la correcta integración de la navegación dentro del entorno de desarrollo SIMATIC WinCC V8.1, se ha implementado un sistema de nomenclatura estandarizado para todos los archivos gráficos de la interfaz (\texttt{.pdl}). Este sistema adopta una estructura jerárquica y funcional, resultando indispensable para la gestión eficiente de proyectos de automatización con un alto volumen de variables y pantallas.

El patrón de diseño definido para nombrar cada interfaz se fundamenta en tres bloques lógicos: \texttt{[Nivel]\_[Categoría]\_[Descripción\_Funcional]} (figura \ref{fig:jerarquia_pantallas}).
\begin{figure}[t]
    \centering
    \includegraphics[scale=0.45]{figuras/jerarquia.png}
    \caption{Ejemplo de jerarquía de pantallas en una interfaz gráfica de supervisión industrial.}
    \label{fig:jerarquia_pantallas}
\end{figure}
\subsubsection{Prefijo Numérico (Jerarquía y Orden de Navegación)}
El primer bloque numérico determina la profundidad de la pantalla dentro del árbol de navegación del SCADA, marcando el flujo operativo:
\begin{itemize}
    \item \textbf{00 (Nivel Raíz):} Reservado exclusivamente para el marco base de la aplicación y la pantalla de inicio principal (\texttt{00\_Main\_Interface}).Físicamente, la arquitectura gráfica se sustenta en esta plantilla maestra que actúa como \textbf{Nivel 0} o marco base.(figura \ref{fig:interfaz_grafica}).
    \item \textbf{1X (Áreas Principales):} Define las divisiones macroscópicas del parque eólico. Incluye la vista general de los aerogeneradores (\texttt{10}), la subestación eléctrica (\texttt{11}), la estación meteorológica (\texttt{12}) y los servicios de diagnóstico del sistema (\texttt{13}, \texttt{14}).
    \item \textbf{2X, 3X, 4X (Niveles de Detalle y Equipos):} Representan submenús y el desglose técnico de los activos. A medida que se incrementa el dígito inicial, aumenta el detalle de la información supervisada. Por ejemplo, los niveles 40, 41 y 42 se destinan a la monitorización de subsistemas de los aerogeneradores, como el generador o el rotor.
    \item \textbf{9X (Pop Ups y Faceplates):} Se reservan para pantallas de uso específico, como ventanas emergentes de alarmas, plantillas de variables (\textit{Faceplates}) o interfaces de configuración avanzada. Estas pantallas no forman parte del flujo de navegación principal, sino que se activan bajo demanda para tareas puntuales.
\end{itemize}

\subsubsection{Identificador de Categoría (Clasificación de Activos)}
El segundo bloque emplea acrónimos técnicos para identificar de forma unívoca la familia de sistemas a la que pertenece la interfaz:
\begin{itemize}
    \item \textbf{WTG (Wind Turbine Generator):} Pantallas dedicadas a la monitorización, control y operación directa de los aerogeneradores.
    \item \textbf{SUB (Substation):} Interfaz enfocada en la infraestructura eléctrica de evacuación, esquemas unifilares y transformadores.
    \item \textbf{MET (Meteorological):} Visualización de los datos climáticos adquiridos por la torre meteorológica.
    \item \textbf{SYS (System):} Funcionalidades globales del entorno SCADA, abarcando la gestión de alarmas y las pantallas de tendencias (\textit{Trends}).
\end{itemize}

\subsubsection{Descripción Funcional}
El tercer bloque explicita el propósito operativo de la pantalla, agilizando su localización por parte del programador o integrador:
\begin{itemize}
    \item \textbf{Overview:} Vista panorámica que agrega los indicadores clave (KPIs) y el estado general de un conjunto de activos.
    \item \textbf{Interface / Detail:} Panel de control principal o plantilla (\textit{Faceplate}) que detalla las variables operativas de un equipo.
    \item \textbf{SLD (Single Line Diagram):} Representación dinámica del esquema eléctrico unifilar.
    \item \textbf{Gen / Rotor / TRF:} Identificadores concretos de subsistemas electromecánicos (Generador, Rotor, Transformador).
\end{itemize}

\vspace{0.5cm}
La adopción de esta nomenclatura trasciende la mera organización de archivos; define la lógica de navegación \textit{Top-Down} requerida en la operación de plantas de generación. El operador inicia la supervisión en una vista global (\texttt{10\_WTG\_Overview}), focaliza la atención en una turbina particular (\texttt{30\_WTG\_Turbine\_Interface}) y profundiza en el diagnóstico de sus componentes mecánicos (\texttt{40\_WTG\_Gen}).\par 

Asimismo, esta estandarización resulta determinante para optimizar el rendimiento de la aplicación gráfica. Al mantener nombres predecibles y estructurados, se facilita el uso de \textit{scripts} y ventanas de imagen (\textit{Picture Windows}) dinámicas, permitiendo que una única pantalla genérica actúe como interfaz para cualquier aerogenerador del parque mediante el paso de parámetros y prefijos de variables estructuradas (\textit{Tag Prefixes}).\par

Para mantener esta filosofía, la navegación del sistema se ha estructurado implementando un diseño centralizado basado en estas ventanas de imagen. Esta técnica de SIMATIC WinCC permite mantener un marco estático de control mientras el área de trabajo cambia dinámicamente. \subsection{Nivel 9: Plantillas dinámicas y reutilización gráfica (\textit{Tag Prefix})}
\label{sec:tag_prefix}

En la monitorización de parques eólicos, se presenta una alta repetición de activos idénticos (decenas de aerogeneradores con la misma instrumentación). Desarrollar un archivo \texttt{.pdl} individual para cada turbina o clúster resultaría ineficiente, consumiendo recursos excesivos del sistema y dificultando el mantenimiento futuro del SCADA. 

Para resolver esta problemática, el diseño de las interfaces de detalle se ha fundamentado en el principio de reutilización gráfica, combinando el uso de las variables de estructura (definidas en el apartado 7.4.3) con la propiedad \textit{Tag Prefix} (Prefijo de variable) de las ventanas de imagen (\textit{Picture Windows}) de SIMATIC WinCC.

La técnica consiste en diseñar una única pantalla genérica (\textit{Faceplate} maestro) donde los objetos gráficos (barras, campos de texto, indicadores de estado) se vinculan a las variables sin especificar la turbina a la que pertenecen (por ejemplo, apuntando a la variable \texttt{.WROT.RotAzi.mag} en lugar de \texttt{WTG1.WROT.RotAzi.mag}).

Durante la operación en el entorno \textit{Runtime}, cuando el operador selecciona un aerogenerador específico desde la vista general, se ejecuta un \textit{script} que realiza dos acciones simultáneas sobre el \textit{Picture Window} central:
\begin{enumerate}
    \item Carga el archivo \texttt{.pdl} de la plantilla genérica.
    \item Asigna el nombre de la turbina seleccionada (ej. \texttt{WTG1}) a la propiedad \textit{Tag Prefix} de la ventana.
\end{enumerate}

Automáticamente, WinCC concatena este prefijo con el sufijo interno de los objetos gráficos, dinamizando la pantalla completa con los datos reales del equipo seleccionado en cuestión de milisegundos. Esta filosofía se ha aplicado intensivamente en los niveles principales del proyecto

\section{Gestión de alarmas y eventos}
\label{sec:alarmas}

La gestión de alarmas y eventos es el componente crítico del sistema SCADA encargado de garantizar la seguridad operativa y la conciencia situacional del operador. Su función principal no es solo la detección de anomalías, sino la jerarquización de las mismas para evitar la saturación de información durante incidentes críticos. 

En este proyecto, la supervisión de eventos se integra de forma transversal mediante el \textit{Picture Window} inferior persistente en la pantalla \texttt{00\_Main\_Interface}. Este diseño asegura que, independientemente del nivel de navegación en el que se encuentre el usuario, las notificaciones de mayor prioridad sean visibles de forma inmediata. La implementación se ha realizado empleando el objeto \textit{WinCC Alarm Control}, configurando ciclos de refresco en tiempo real y almacenamiento en una base de datos SQL histórica para auditorías posteriores.

\subsection{Criterios de señalización: Norma IEC 60073}

Para garantizar una interpretación unívoca de los estados del proceso, se ha seguido la normativa internacional \textbf{IEC 60073} (\textit{Principios básicos y de seguridad para las interfaces hombre-máquina, el marcado y la identificación}) \cite{iec60073}. Esta norma establece una codificación cromática estricta basada en el significado de la seguridad y el estado del equipo:

\begin{itemize}
    \item \textbf{Rojo:} Indica una situación de peligro, emergencia o actuación inmediata.
    \item \textbf{Amarillo/Ámbar:} Señala una condición anormal o un cambio crítico de estado que requiere precaución o supervisión.
    \item \textbf{Verde:} Representa una condición normal de funcionamiento o seguridad.
    \item \textbf{Azul:} Indica una condición de carácter obligatorio o una intervención específica del operador.
    \item \textbf{Blanco/Gris:} Utilizado para estados neutros o falta de información específica de seguridad.
\end{itemize}

\subsection{Configuración e implementación del sistema de avisos en SIMATIC WinCC}
\label{sec:configuracion_alarmas_wincc}

La materialización de la gestión de alarmas en el entorno \textit{Runtime} se estructura mediante el subsistema \textit{WinCC Alarm Logging}. Para mantener la coherencia con el modelo de información estandarizado (Capitulo \ref{cap:modelo_informacion}), todos los eventos discretos del aerogenerador se han concentrado en el Nodo Lógico de alarmas, denominado \texttt{WALM} (ver Anexo \ref{cap:Tablas_Variab}).

La categorización de estos eventos determina su gravedad y a consecuencia su color en la interfaz gráfica, regulando la interacción mediante el bloqueo, acuse y reconocimiento.

\subsubsection{Categorización funcional}

Las señales de error, configuradas en WinCC como variables de tipo \textit{Single Point Status (\ac{SPS})}, se clasifican en tres clases fundamentales de avisos:

\begin{itemize}
    \item \textbf{Alarmas de Emergencia (Críticas):} Eventos que implican un riesgo inminente para la integridad de la máquina o de la red, provocando la parada inmediata de la turbina. 
    \item \textbf{Alarmas de Fallo (Operativas):} Indican averías de componentes mecánicos o eléctricos que interrumpen la producción, pero mantienen el sistema en un estado controlado. 
    \item \textbf{Advertencias (Avisos):} Notifican condiciones ambientales o desviaciones técnicas que requieren atención preventiva sin forzar una parada inmediata.
\end{itemize}

\subsubsection{Matriz de estados visuales y filosofía de acuse}

El comportamiento cromático en el objeto \textit{WinCC Alarm Control} sigue una lógica secuencial que muestra si un evento ha sido reconocido por el operador del centro de control, manteniendo la coherencia con los principios de señalización de la norma IEC 60073. 
\par
Cuando una alarma entra en el sistema (estado activo sin acusar), la interfaz prioriza la captura de la atención del usuario: el texto se presenta en color blanco sobre un fondo cromático saturado representativo de su gravedad (rojo para emergencias y fallos, o amarillo/ámbar para advertencias).\par

En el momento en que el operador realiza el acuse de recibo (\textit{Acknowledgment}), si la condición de fallo persiste en el campo (estado activo acusado), la codificación visual invierte los colores para indicar un estado de control humano: el fondo de la línea de alarma pasa a ser completamente blanco y el texto adopta de forma fija el color identificativo de su clase, cesando todo parpadeo. \section{Integración de IIoT en la solución}

En el contexto actual de la Industria 4.0, la monitorización y supervisión de infraestructuras energéticas distribuidas, como es el caso de un parque eólico, se demandan soluciones que vayan más allá de los sistemas SCADA tradicionales aislados en planta. La convergencia entre las Tecnologías de la Operación (OT) y las Tecnologías de la Información (IT) permite habilitar el ecosistema del Internet Industrial de las Cosas (IIoT), facilitando el análisis de datos a gran escala, la accesibilidad remota y la escalabilidad del sistema.\par

Este capítulo describe en detalle el diseño, despliegue e integración de una arquitectura en la nube orientada a la adquisición y visualización de datos  provenientes de los aerogeneradores. Mientras que el sistema local (basado en SIMATIC WinCC V8.1) se encarga de la adquisición en tiempo real (Edge) y el control a nivel de campo mediante protocolos industriales como \ac{OPC UA}, la arquitectura propuesta en este capítulo tiene como misión exportar estos datos de manera segura y eficiente hacia la nube para su almacenamiento histórico y explotación visual.\par

Para lograr un sistema robusto, modular y replicable, se ha diseñado una arquitectura basada en microservicios contenerizados, apoyada en las siguientes capas lógicas (figura \ref{fig:IIoT}):

\begin{itemize}
    \item Capa de Adquisición y Emisión (Edge): Representada por el módulo Cloud Connect del SCADA WinCC, encargado de empaquetar las variables de proceso (velocidad del viento, potencia generada, estados, etc.) y publicarlas mediante el protocolo de mensajería ligero MQTT.
    \item Capa de Infraestructura (Cloud): Soportada por una instancia virtual (EC2) alojada en Amazon Web Services (AWS), que proporciona el entorno computacional necesario con alta disponibilidad. El despliegue de los servicios se ha orquestado íntegramente mediante Docker, garantizando el aislamiento de los procesos y facilitando futuras migraciones.
    \item Capa de Ingesta y Procesamiento Intermedio: Implementada con Node-RED, que actúa como cliente MQTT para la suscripción a los tópicos de la planta, y proporciona un entorno de flujo de datos visual para filtrar y transformar los paquetes JSON recibidos antes de su inserción en la base de datos.
    \item Capa de Almacenamiento (TSDB): Soportada por InfluxDB, una base de datos de series temporales optimizada para la escritura de alta frecuencia y el almacenamiento eficiente de datos con marcas de tiempo (timestamps).
    \item Capa de Visualización: Desarrollada sobre Grafana, que consulta directamente a la base de datos para construir dashboards interactivos, permitiendo la monitorización del rendimiento del parque eólico desde cualquier ubicación con acceso a internet.
\end{itemize}

\subsubsection{Orquestación de microservicios mediante Docker}

Para garantizar la portabilidad y escalabilidad de la solución, se ha optado por contenedorizar cada componente de la arquitectura (Node-RED, InfluxDB y Grafana) utilizando Docker, el cual permite empaquetar las aplicaciones junto con sus dependencias en contenedores ligeros que abstraen el sistema operativo sobre el que se ejecutan. De esta forma, las configuraciones especificas de diferentes entornos donde se puedan desplegar los servicios (desarrollo, pruebas, producción) se gestionan de forma consistente y replicable.\par

\begin{itemize}

    \item Persistencia de Datos (Docker Volumes): Dado que los contenedores son efímeros por naturaleza, se han configurado volúmenes vinculados al almacenamiento físico de la instancia EC2. Esto garantiza que, ante un reinicio del sistema o una actualización del software, los datos históricos del parque eólico y las configuraciones de los dashboards no se pierdan.
\end{itemize}

\section{Ciberseguridad IEC 62443-4-2:2019}
Esta sección es una directriz de seguridad válida para SIMATIC HMI WinCC V8.1 que describe sus funciones de ciberseguridad. Sigue la norma \textbf{IEC 62443-4-2:2019} de ciberseguridad para sistemas de automatización y control industrial. La norma IEC 62443 define cuatro niveles de seguridad (SL, por sus siglas en inglés) distintos, basados en los recursos, las habilidades y la motivación del posible atacante \cite{iec62443-4-2}:

\begin{itemize}
    \item \textbf{SL 1:} Protección contra vulneraciones casuales/accidentales:
    Protección contra el acceso no autorizado por parte de alguien sin intención maliciosa.
    \item \textbf{SL 2:} Protección contra vulneraciones intencionadas con medios sencillos:
    Protección contra ataques genéricos y de bajos recursos.
    \item \textbf{SL 3:} Protección contra ataques sofisticados:
    Protección contra atacantes (hackers) con recursos moderados, motivación moderada y conocimientos específicos sobre Sistemas de Automatización y Control Industrial (IACS, por sus siglas en inglés).
    \item \textbf{SL 4:} Protección contra ataques altamente sofisticados / de Estados-nación:
    Protección contra atacantes con amplios recursos, alta motivación y profunda experiencia en IACS.
\end{itemize}

Los requisitos fundacionales (FR) son los requisitos de seguridad que deben cumplirse para alcanzar  el SL específico. Estos requisitos se dividen en siete categorías principales, cada una con sub-requisitos específicos. A continuación se describen los requisitos fundacionales para cada nivel de seguridad:\par
\subsection{FR1: Identificación y autenticación de usuarios (IAC)}
\begin{enumerate}

    \begin{itemize}
        \item CR1.1: Acceso limitado a aplicaciones de WinCC V8.1 mediante grupos de usuarios de Windows. Acceso restringido de datos del runtime mediante grupos de usuarios de WinCC. (SL 1)
        \item RE 1:Identificación única de usuarios mediante ID única, creación de roles con permisos específicos. (SL 2)
        \item RE 2(No ofertado): Autenticación multifactorial para todas las interfaces (SL 3-4).
    \end{itemize}
    
    Nivel de seguridad SL-T 2 : Requerimientos CR1.1 + RE 1 .\par

    \item[] \textbf{Identificación y autenticación de dispositivos (CR1.2)}
    
    \begin{itemize}
        \item CR1.2:    Identificación y autenticación de dispositivos básica(SL 2)
        \item RE 1: Identificación única de dispositivos mediante certificados digitales o credenciales seguras, Integrado en comunicaciónes \ac{OPC UA}. (SL 3-4)
       
    \end{itemize}

    Nivel de seguridad SL-T 4: Requerimientos CR1.2 + RE 1.\par

    \item[] \textbf{Gestión de cuentas (CR1.3)}
    
    \begin{itemize}
        \item CR1.3:  Gestión de cuentas mediante:  1-User Administrator, 2-WinCC UserAdminControl, 3-SIMATIC Logon. (SL 1-4).

    Nivel de seguridad SL-T 4: Requerimientos CR1.3.\par

     \item[] \textbf{Gestión de autenticación (CR1.5)}

       \item RE 1(No ofertado): Seguridad de hardware para autenticación (SL 3-4).
    \end{itemize}
    Nivel de seguridad SL-T 2: Requerimientos CR1.5.\par

       \item RE 2(No ofertado): Caducidad de contraseñas para dispositivos, humanos y software (SL 4).
    \end{itemize}
    Nivel de seguridad SL-T 2: Requerimientos CR1.7.\par

    \item[]  \textbf{Infraestructura de clave pública (CR1.8)}

    \end{itemize}
    Nivel de seguridad SL-T 4: Requerimientos CR1.8.\par
    
  \item[] \textbf{Retroalimentación de autenticación (CR1.10)}

    \end{itemize}
    Nivel de seguridad SL-T 4: Requerimientos CR1.10.\par
 \item[] \textbf{Autenticación fallida (CR1.11)}
    
    \begin{itemize}
        \item CR1.11: Limitación del número de intentos de autenticación fallida antes de bloqueo.(SL 1-4).
    \end{itemize}
    Nivel de seguridad SL-T 4: Requerimientos CR1.11.\par

    \end{itemize}
